How to configure this test

Use this to check whether your numeric columns are normally distributed before running tests that assume normality. Drag one or more numeric columns — such as post_score — into the Variable(s) slot, and you'll get a histogram, a Q-Q plot, and normality tests (Shapiro-Wilk and K–S) for each. Use this when you have many numeric variables that overlap and you want to boil them down into a few underlying components — e.g. several survey items measuring the same idea. Drag all the numeric variables you want to summarise (x1–x5) into Variables; you need at least two. Keep standardize on so variables on different scales count equally, then choose how many components to keep with n components. Categorical columns like region can't go in — only numeric measurements work.

How to configure this test

Use this for variance-based SEM — well suited to prediction, smaller samples, or formative constructs. Build the model on the same canvas as CB-SEM: drag each construct's items together, then draw arrows for the relationships you hypothesize (including any mediation — chained paths get an Indirect effects table automatically; moderation is planned for a later version). Switch a construct to formative when its items cause the construct rather than reflect it. Bootstrapping gives the significance of the path coefficients.

Produces — tables: outer loadings/weights, reliability & validity (CR/AVE/ρA), HTMT discriminant validity, structural paths (β, f²), structural model quality (R² / Q²), indirect effects (when paths chain) figure: path diagram

How to read this page:
Every selected test is drawn as a full configuration card with the same parts — ① drag columns into roles, ② set options, ③ a "How to configure this test" guide, ④ what it produces.
The dashed amber "How to configure this test" box is a configure-time guide — separate from the results-screen "How to read this test" layer in the main spec. One teaches setup; the other teaches interpretation.
Latent-variable models (CB-SEM, PLS-SEM, AVE, CR) replace the role slots with a model-building canvas where the student drags items into constructs and draws the paths.
Drawn grey reasons that lean on the example's meaning (e.g. “belongs to a different construct”, “measured once, not across time”) are illustrative — the app's actual greying is computed from level / arity constraints only, and semantic grouping (which columns are repeated measures of one variable, or items of one construct) is declared by the student's own drag placement, never inferred.
Slot tags like count, datetime / ordered, id and nom·nested are auto-detected column properties used for slot eligibility — not step-4 measurement levels the student assigns (those stay nominal · ordinal · interval · ratio).
