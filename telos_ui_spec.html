Results — the "how to read" layer

  • Generic per test type. Written and reviewed once, then stored as static content. It is not computed from the student's data, so there is nothing to hallucinate or get wrong at run time.
  • Teaches the general reading — what each statistic means, what a significant vs. non-significant result indicates, and the usual effect-size conventions.
  • Never interprets the student's specific result, and never points to specific cells or values in their output — edge cases and data-specific issues could send them in the wrong direction.
  • Identical everywhere. The exact same text appears in the PDF, the LaTeX, and the on-screen preview.

Results — bundle (.zip)

telos-results/ ├─ report.pdf ├─ report.tex ├─ analysis.R ├─ data/ │ └─ cleaned.csv ← only when R script is selected ├─ 01_independent-t-test/ │ ├─ table_group-statistics.png │ ├─ table_t-test.png │ └─ figure_boxplot.png ├─ 02_one-way-anova/ │ ├─ table_anova.png │ ├─ table_tukey-posthoc.png │ └─ figure_means-plot.png └─ 03_cb-sem/ ├─ table_efa-loadings.png ├─ table_cfa-loadings.png ├─ table_fit-indices.png ├─ table_structural-paths.png └─ figure_path-diagram.png
  • One .zip whenever the export is more than one file (multiple formats, multiple tests, or image folders) — never N separate downloads. A single one-file export (e.g. only the PDF) downloads directly, no .zip. (Tree above shows everything ticked.)
  • Each selected test gets its own folder, prefixed NN_ in configured order; files inside are labelled by content (table_*, figure_*).
  • report.pdf / report.tex / analysis.R sit at the root.
  • data/cleaned.csv rides along whenever the R script is selected, so analysis.R runs via a relative path on the student's own machine — not just on the server.
  • A folder or file appears only if its format was ticked — per-test image folders require table and/or figure images; each report file requires its own tick.

Results — engine & formats (our side)

  1. Each test produces one canonical result object — { metadata · table data · figure spec · how-to-read text }.
  2. Objects are assembled in configured order into a single Quarto .qmd.
  3. quarto render → report.pdf (APA 7).
  4. The .tex is produced as a byproduct of that same render → report.tex.
  5. knitr::purl() extracts the executable code → analysis.R — literally the code that made the PDF, so it reproduces it exactly.
  6. Figures saved as PNG via ggplot2; APA tables saved as PNG via gt / flextable.
  7. Everything (plus cleaned.csv if R was selected) is zipped per the structure above.
report.pdf report.tex analysis.R table_*.png figure_*.png
Tables render in APA 7th (gt / flextable); figures render in ggplot2.
Reproducibility guarantee — because all five outputs derive from one render, the R script, LaTeX, and PDF can't drift apart.
Cost — figures are styled by the R/ggplot theme rather than the web theme, and the backend container ships Quarto + TinyTeX + the R packages.
