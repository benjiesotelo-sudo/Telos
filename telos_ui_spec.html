3 · Terms guide
Type vs level · the 4 levels · missing-data handling
DRAFT copy — for review

“Type vs level: the type is how a column is stored (numbers, text, dates); the level is what its values mean for analysis — you set the level. Nominal — named categories with no order (gender, region). Ordinal — ordered categories with uneven steps (a 1–5 satisfaction rating). Interval — equal steps but no true zero (temperature in °C). Ratio — equal steps and a true zero (income, test score). Missing data: empty cells — by default each test simply skips the rows that are incomplete for its own columns (step 4a lets you change this).”

Results — the "how to read" layer

  • Generic per test type. Written and reviewed once, then stored as static content. It is not computed from the student's data, so there is nothing to hallucinate or get wrong at run time.
  • Teaches the general reading — what each statistic means, what a significant vs. non-significant result indicates, and the usual effect-size conventions.
  • Never interprets the student's specific result, and never points to specific cells or values in their output — edge cases and data-specific issues could send them in the wrong direction.
  • Identical everywhere. The exact same text appears in the PDF, the LaTeX, and the on-screen preview.

  • One .zip whenever the export is more than one file (multiple formats, multiple tests, or image folders) — never N separate downloads. A single one-file export (e.g. only the PDF) downloads directly, no .zip. (Tree above shows everything ticked.)
  • Each selected test gets its own folder, prefixed NN_ in configured order; files inside are labelled by content (table_*, figure_*).
  • report.pdf / report.tex / analysis.R sit at the root.
  • data/cleaned.csv rides along whenever the R script is selected, so analysis.R runs via a relative path in the student's own R/RStudio — the same analysis, reproduced on their machine.
  • A folder or file appears only if its format was ticked — per-test image folders require table and/or figure images; each report file requires its own tick.

Results — engine & formats (our side)

  1. All statistics run in the browser — WebR (R compiled to WebAssembly) in a worker; there is no backend, and the uploaded data never leaves the browser.
  2. R packages preload from the WebR repo and cache in the browser, so repeat visits skip the download.
  3. Each test produces one canonical result object — { metadata · table data · figure spec · how-to-read text }.
  4. Objects are assembled in configured order into the results preview, the report, and analysis.R.
  5. Figures are ggplot2, rendered inside WebR and returned as PNG; APA tables saved as PNG.
  6. analysis.R is a reproducibility script — the student can run it themselves in R/RStudio (with data/cleaned.csv) to reproduce every number and figure.
  7. report.pdf / report.tex are generated in-browser (the exact PDF route is a deferred build-time choice); everything is zipped per the structure above.
report.pdf report.tex analysis.R table_*.png figure_*.png
Tables render in APA 7th; figures render in ggplot2, inside WebR, to PNG.
Number formatting (APA) — test statistics (t, F, χ², z) to 2 dp · p to 3 dp, no leading zero, floored at "<.001" · df as an integer when mathematically integer (pooled t, chi-square), 2 dp otherwise (Welch) · M / SD to 2 dp in tables, 1 dp in the fill-in APA sentence (per the exemplar) · effect sizes (d) to 2 dp with a leading zero · CIs in square brackets [lower, upper] — DRAFT — confirm: the exemplar prints CI bounds to 1 dp while means use 2 dp; matched here.
Reproducibility guarantee — analysis.R is generated from the same registry R code the WebR engine ran, so the script, the report, and the on-screen results can't drift apart.
Cost — figures are styled by the R/ggplot theme rather than the web theme, and the first visit downloads the WebR (WASM) runtime + packages, cached for repeat visits — there is no backend to ship or run.
